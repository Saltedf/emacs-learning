# Module 7: 写一个真实的包

> **目标**: 设计、实现、测试、文档化、发布一个真实可用的包到 MELPA
> **时长**: 3 周 (~60 小时)
> **难度**: ★★★★☆
> **依赖**: Module 6 (能写 major mode)
> **核心产出**: 你的包被 MELPA 合并

---

## 0. 这个模块在做什么

Module 1 到 6 教你"用 Emacs"和"改造自己的 Emacs"——所有代码都是私人定制,跑在你机器上就够了。Module 7 是一个分水岭:你从这里开始**为社区写代码**。这意味着思维方式的根本转变。私人代码只要"在我机器上 work"就行;社区代码要满足一群陌生人能装、能用、能信任、能扩展。

你已经写过 my-todo (Module 3) 和 mylang-mode (Module 6)。但它们有几个共同的"私房菜"特征:

- 可能没有完整文档——只有你自己知道怎么用
- 可能没有 ERT 测试——下次你改了一行,可能默默破坏功能而不知道
- 不在 MELPA,别人用不了——你的成果无法被复用、被审视、被改进
- header 不规范,装到别人 Emacs 可能冲突或加载失败
- 没有 `;;;###autoload`,启动慢

Module 7 教你**把代码升级到生产级别,发布到社区**。"生产级别"不是空话——它有一组具体的、可检验的属性:任意一个 Emacs 用户 `M-x package-install` 你的包,装完之后你的所有公开功能都能用、README 看得懂、bug 能用 ERT 重现、跨多个 Emacs 版本不挂。这五点就是 Module 7 的核心检查表。

为什么这个转变重要?因为 Emacs 的生态本质上是**社区驱动的包仓库**。Emacs 本身的核心代码 (~1.5M 行 C + Elisp) 不会快速演化,真正让 Emacs "现代"的是 magit、org、lsp-mode、vertico 这些社区包。你加入这个生态,就从"消费者"变成"生产者"。这是一种身份转变,也是一种责任转变——你写的代码会被陌生人依赖,所以你要对自己更严格。

## 1. 完整流程

下面是从"想法"到"MELPA 上线"的完整管线。每一步都不是可选项,跳过任何一步都会在后面反弹回来咬你。

```
1. 选题: 找一个真实痛点
2. 调研: 现有方案
3. 设计: 命名、API、特性
4. 实现: TDD
5. 测试: ERT,多版本 Emacs
6. 文档: README + Texinfo
7. CI: GitHub Actions
8. 发布: GitHub
9. MELPA: 提 PR
10. 维护: 处理 issue, PR
```

这十步看似线性,实际上是**循环**:维护阶段你会发现问题,回头改设计、加测试、补文档,然后发布新版本。**写包不是一次性工作,是一个长期承诺**。这就是为什么"选题"是第一步——如果你选了你根本不关心的痛点,3 个月后没人维护,你的包就变成 MELPA 上的死代码,拖累整个生态。

每一步对应本模块的一个专题文件:步骤 1-3 在 `package-design.md`,步骤 4-5 在 `ert-testing.md`,步骤 6 在 `docs-texinfo.md`,步骤 7-9 在 `melpa-submission.md`,步骤 10 散落在所有文件但 `capstone.md` 会强调。

## 2. 模块内容

### 2.1 设计 (`package-design.md`)

讲**怎么想清楚要做什么**,而不是怎么写代码。这是新手最容易跳过的一步——很多人直接开 editor 开始写,3 天后发现自己做的东西和现有包重复,或者 API 设计得很烂无法扩展。这一节教你"先想后写":选题的过滤标准、命名约定、公开 vs 内部 API 的边界、状态管理、hook 设计、错误处理。

### 2.2 测试 (`ert-testing.md`)

ERT 是 Emacs 内置的测试 framework。这一节从第一性原理讲**为什么需要测试**(代码会变,变可能引入 bug,bug 影响用户,所以需要自动化回归),然后讲 ERT 的所有用法:基本 assertion、数据驱动测试、buffer 测试、font-lock 测试、mock/cl-letf、CI 配置。讲完后你会知道怎么写出一个"敢放心改"的包。

### 2.3 文档 (`docs-texinfo.md`)

文档不是事后补的,是设计的一部分。这一节讲两类文档:**README**(GitHub 看到的门面)和 **Texinfo 手册**(Emacs 内 `C-h i` 看到的深度文档)。两者的受众不同——README 给"决定要不要装"的人看,手册给"已经装了在用"的人看。优秀的包两者都有。

### 2.4 MELPA (`melpa-submission.md`)

把代码变成"全世界能装"。讲 MELPA 是什么、为什么是社区事实标准、recipe 怎么写、PR 怎么提、review 怎么处理。也对比 NonGNU ELPA (FSF 官方的替代品)。

### 2.5 Capstone

`capstone.md` 是三周的实战项目:把 Module 3 或 6 的代码升级到 MELPA-ready,真的发布出去。没有这一步,前面学的都是纸上谈兵。

## 3. 选题灵感

选题是写包最难的一步,因为代码可以照着教程写,但"选什么"完全靠你自己观察。下面是几条选题的"心法"。

### 3.1 寻找痛点

最好的选题来自你**自己的痛点**——你自己用 Emacs 时反复觉得烦的事。这种选题有几个好处:第一,你天然是用户,知道需求;第二,你不会因为"没动力"半途而废;第三,你能在自己日常使用中持续验证设计。

观察你自己用 Emacs 时,问三个问题:

- 反复做的操作? → 写命令自动化。比如你每次改完代码都手动跑 lint,就可以写个 `my-lint-on-save` minor mode
- 觉得麻烦的事? → 写 minor mode 改进。比如某个 mode 的快捷键反人类,你可以 wrap 它
- 想要但不存在的功能? → 写包。这是最高价值但也最难,因为要确认"真的不存在"

第三点尤其要警惕。**"我觉得应该有"不等于"真的没有人做过"**。在动手之前,先在 MELPA、NonGNU ELPA、Emacs Wiki、GitHub 搜一遍——大概率有人已经做过 80%,你可以 fork 改,或者贡献 PR,而不是从零开始。

### 3.2 不要做

三条红线:

- **已有更好的包**:又一个 magit?又一个 org-mode?又一个 vertico?这种选题几乎注定失败,因为你的小作坊打不过社区已经打磨了 5 年的成熟作品。除非你有**根本性不同的设计**(比如 magit 是 porcelain,你做一个基于 Ivy 的 minimal porcelain),否则别浪费时间。
- **太复杂**(一周写不完):新手最容易高估自己。500-2000 行的包是合适的规模——再大就要拆成多个包,或者需要多人协作。
- **太小众**(用户少):不是说不写,而是要清醒——小众包可能 1 年只有 5 个 star,你的投入产出比很差。除非纯为学习,否则挑一个"至少 100 人会用到"的痛点。

### 3.3 好的选题

一个"好选题"的画像:

- **小工具**(500-2000 行)——你能 hold 住全部代码
- **解决特定问题**——一句话能说清楚做什么
- **API 清晰**——3-5 个公开函数就够了
- **测试覆盖好**——核心逻辑 90%+ 覆盖

好的选题实例(都是真实存在、规模合适、用户喜欢的包):

- **主题切换器**:管理多个 color theme,根据时间/项目自动切。例:`circadian`、`auto-darken`
- **模板系统**(替代 yasnippet 的轻量版):简单的 keyword expansion。例:`tempel`
- **项目导航**:在项目内快速跳文件/符号。例:`projectile`(已是巨头)、`project-tab-groups`
- **命令运行器**:定义一组项目命令,一键跑。例:`quickrun`
- **配置管理**:跨机器同步配置。例:`chemicmacs`
- **文本变换**:批量改字符串、表格、CSV。例:`string-inflection`
- **与外部工具集成**:git/docker/k8s wrapper。例:`magit`、`docker-tramp`

注意这些例子——**每个都是真实存在的、活跃的、有人维护的包**。你写一个类似的,就算 100 行的小工具,只要解决真痛点,也会有几十个 star。

## 4. 毕业检查

### 概念题

1. defgroup + defcustom 在包里干嘛?
2. `;;;###autoload` 怎么用?
3. ERT 测试的基本结构?
4. MELPA recipe 是什么?
5. CI (GitHub Actions) 怎么配?

这五个问题覆盖了 Module 7 的核心概念。如果有一个答不上来,回去读对应的小节——不要带疑问进 capstone,因为 capstone 是 21 天的实战,卡壳一天就是 4% 进度。

### 实操

1. 写一个完整包(≥ 500 行)
2. ≥ 10 个 ERT 测试
3. README 完整
4. 提交 MELPA

实操要求是**最小可行性发布**——再小都可以,但必须走完全流程。这是关键:**完成比完美重要**。一个 500 行、10 个测试、README 简单、PR 合并的包,远胜于一个 5000 行、80 个测试、文档精美但永远没发布的"完美作品"。完成会让你真正理解整个生态,完美会让你停留在幻想里。
